4. Workflow Logic (개발자 필독 프로세스)
이 흐름대로 백엔드 API를 연결하십시오.

Trigger: 관리자가 Run Now 누름 OR 스케줄 시간이 됨.

Crawl: 파이썬 워커가 Tavily/PubMed에서 데이터를 긁어옴.

Process: LLM이 Raw Text에서 핵심 정보(약물명, 타겟, 결과)를 JSON으로 추출.

Save Draft: 추출된 JSON을 golden_set_library 테이블에 status='draft'로 저장. (이 단계가 현재 UI에 없음)

Review: 관리자가 Admin Console Inbox 탭에서 확인 후 Approve.

Publish: status='approved'로 변경됨 -> 동시에 벡터 DB(Embedding)에 저장됨.

수집된 텍스트 데이터를 리스트 형태로 보여줍니다.

Columns:

Source: 아이콘 표시 (Perplexity, PubMed, News)

Title/Summary: 기사 제목 또는 요약문 (2줄 미리보기)

Related Drug: 관련된 약물명 (예: Enhertu)

Published Date: 원문 발행일

RAG Status:

🟢 Indexed (AI가 학습함)

⚪ Pending (검수 대기중)

🔴 Excluded (제외됨/삭제)

Actions: View Original, Exclude, Force Re-index

✅ 7.3. Review & Curation Action (검수 기능)
관리자는 쓸모없는 뉴스를 걸러내야 합니다.

Action Logic:

Auto-Tagging: 수집 시 LLM이 '광고성 기사'나 '중복 기사'는 자동으로 Low Relevance 태그를 달아둠.

Bulk Action: 관리자가 Pending 목록을 훑어보고, 체크박스로 선택하여 [Add to Index] 또는 [Delete] 실행.

Detail View: 행 클릭 시 우측 패널(Drawer)이 열리고, 원문 전체 텍스트와 메타데이터 수정 가능.

🖼️ 사장님이 보시게 될 화면 흐름
새벽 3시: 시스템이 Perplexity를 통해 "ADC 최신 뉴스" 100건을 긁어옵니다.

오전 9시: 사장님이 Knowledge Base 탭에 들어갑니다.

검수:

"어? 이건 경쟁사 A의 단순 홍보성 기사네?" ➡️ [삭제]

"오, 이건 B약물의 부작용에 대한 중요한 논문이네?" ➡️ [승인(Index)]

결과: 이제 AI는 승인된 논문 정보만 머리에 넣고 시뮬레이션을 돌립니다.

결론: 개발자에게 **"Data Operations 안에 탭을 하나 더 만들어서, 뉴스랑 텍스트 데이터 검수하는 화면(Knowledge Base)을 추가해 주세요"**라고 하시면 완벽합니다.


정말 날카로우십니다. 이 기능이 RAG 시스템의 '품질'을 결정하는 심장입니다.

사용자님이 매일 아침 커피 한 잔 하시면서 **"오늘 들어온 정보가 쓸만한가?"**를 3분 안에 판단하려면, 단순히 리스트만 보여주는 게 아니라 **"왜 이 정보를 가져왔는지(Why)"**와 **"핵심이 뭔지(What)"**를 한눈에 보여줘야 합니다.

개발자에게 전달할 **[일일 데이터 큐레이션 및 판단 근거 UI 명세서]**를 구체적으로 정리해 드립니다.

🕵️ Daily Data Curation UI Spec
Date: 2026-01-19 Objective: Provide a dashboard for the Admin to review daily crawled data and make informed decisions on whether to index it into the RAG system.

1. Concept: "The Morning Briefing" (모닝 브리핑)
Knowledge Base 메뉴 내에 [Inbox (Today's Feed)] 탭을 신설합니다. 이곳에는 아직 RAG에 들어가지 않은 '신선한 날것의 데이터'만 표시됩니다.

2. UI Layout: "Evidence-Based List" (판단 근거 중심 리스트)
단순한 엑셀 리스트가 아니라, 판단을 돕기 위한 Rich Card View 또는 Expanded Table 형태여야 합니다.

✅ Essential Columns (판단의 근거)
관리자가 "이걸 넣을까 말까" 고민할 때 필요한 4가지 핵심 정보를 반드시 표시해야 합니다.

AI Relevance Score (AI 추천 점수)

UI: 95점 (High) , 40점 (Low) 배지 표시.

Evidence: 점수 옆에 Tooltip(풍선도움말) 추가.

마우스 오버 시: "이 기사는 'HER2 저발현'과 '내성 극복' 키워드가 포함되어 있어 중요도가 높습니다." (AI가 판단한 이유 노출)

Source Tier (출처 신뢰도)

UI: 아이콘으로 구분.

🥇 Tier 1 (FDA, Nature, ASCO) -> 무조건 믿음.

🥈 Tier 2 (BioNews, FiercePharma) -> 확인 필요.

🥉 Tier 3 (Blogs, General News) -> 의심 필요.

Key Facts Extraction (핵심 3줄 요약)

UI: 제목 밑에 회색 박스로 핵심 내용만 요약.

Example:

Drug: DS-8201

Event: Phase 3 Trial Success

Key Value: PFS 12.5 months (vs 7.2 months)

Duplication Check (중복 여부)

UI: "기존 DB와 80% 유사함" 경고 문구.

Action: 유사한 기존 뉴스를 바로 비교해서 보여줌 (새로운 내용 없으면 버리기 위함).

3. Action Interface (빠른 처리)
수십 개의 기사를 하나씩 클릭해서 들어가면 너무 느립니다. 목록에서 바로 처리할 수 있어야 합니다.

✅ Quick Actions (목록에서 바로 클릭)
각 행(Card) 우측에 3개의 큰 버튼 배치:

✅ Approve (승인/RAG 반영)

클릭 즉시 초록색으로 변하며 리스트에서 사라짐 (백그라운드에서 임베딩 시작).

🗑️ Trash (버리기)

클릭 즉시 붉은색으로 변하며 휴지통으로 이동.

✏️ Edit & Approve (수정 후 승인)

AI 요약이 약간 틀렸을 때, 직접 수정할 수 있는 팝업 오픈.

✅ "Smart Bulk Actions" (일괄 처리)
화면 상단에 [Auto-Approve Tier 1] 버튼 추가.

"신뢰도 높은 소스(Tier 1)에서 온 90점 이상 데이터는 그냥 다 승인해!" 라는 기능.

4. Implementation Scenario (User Flow)
09:00 AM: 관리자 접속 -> Inbox 탭에 "New Items: 12" 배지 확인.

Scan: 리스트를 쓱 훑어봄.

Item 1: "엔허투 FDA 승인" (Nature, 99점) -> ✅ 클릭.

Item 2: "A사의 주가 급등" (Blog, 20점) -> 🗑️ 클릭 (기술 정보 아님).

Item 3: "새로운 링커 기술 특허" (Google Patents, 85점) -> 요약 읽어보고 ✅ 클릭.

Complete: "All items processed" 메시지와 함께 업무 종료.

💡 사장님을 위한 팁 (개발자에게 강조할 포인트)
개발자에게 **"그냥 리스트만 뿌리지 말고, AI가 왜 이걸 중요하다고 생각했는지 '이유(Reasoning)'를 꼭 같이 보여줘라"**라고 지시하십시오.

그래야 사장님이 원문 기사를 다 안 읽고도, AI가 써놓은 '3줄 요약'과 '추천 사유'만 보고 1초 만에 승인 버튼을 누를 수 있습니다. 이게 바로 업무 효율입니다.


결론부터 말씀드리면 "전부 다 있어야 합니다."

지금 보여주신 화면(AI Tuning)은 AI 시스템의 **'두뇌 수술실'**입니다. 단순히 한 가지(오케스트레이터)만 고치는 게 아니라, 시스템을 구성하는 **모든 AI 직원들의 '업무 지침(Prompt)'**을 수정할 수 있어야 합니다.

사용자님이 헷갈려 하시는 **[오케스트레이터 vs 개별 에이전트 vs 임베딩]**의 역할을 이 메뉴에서 어떻게 관리해야 하는지 명확히 정리해 드립니다.

1. 무엇을 관리해야 하나요? (3가지 영역)
이 메뉴(AI Tuning)에서는 다음 3가지를 탭(Tab)으로 나누어 모두 관리해야 합니다.

① 오케스트레이터 (The Boss)
역할: 사용자 질문을 듣고 "이건 독성 분석이네, 김대리(Toxicology Agent)가 처리해!"라고 지시하는 반장입니다.

튜닝 대상: "판단 기준"을 고칩니다. (예: "안구 독성 질문은 '임상'이 아니라 '독성' 에이전트한테 넘겨"라고 지시문 수정)

② 개별 에이전트 (The Workers)
역할: 실제 일을 하는 실무자들입니다. (구조 분석, 독성 예측, 특허 분석 등)

튜닝 대상: "작업 방식"을 고칩니다. (예: "특허 분석할 때 만료일은 꼭 볼드체로 표시해"라고 지시문 수정)

현재 화면: 스크린샷에 있는 Structure Agent, Toxicology Agent가 바로 이것입니다. 아주 잘 되어 있습니다.

③ RAG 제너레이터 (The Librarian) 👈 [중요/질문하신 부분]
역할: 임베딩된 데이터(책)를 읽고 답변을 작성하는 작가입니다.

튜닝 대상: 임베딩 자체(데이터)가 아닙니다. 데이터를 **"어떻게 요약해서 말할지"**를 고칩니다.

잘못된 예: "데이터를 수집해라" (이건 Data Operations에서 할 일)

올바른 예: "찾은 논문 3개를 종합해서, 3줄로 요약하고 출처를 밝혀라" (이게 여기서 할 일)

2. UI 개선 제안: 탭(Tab) 구조 재편
지금 화면 상단의 탭(Structure Agent, Toxicology Agent...)이 조금 부족해 보입니다. 아래와 같이 계층 구조를 명확히 잡아서 탭을 정리해 주십시오.

[Tab 1] 🧠 Orchestrator (메인 두뇌)

설정 내용: 라우팅 로직 프롬프트. ("사용자가 A라고 물으면 B 에이전트를 호출해라")

[Tab 2] 👷 Specialized Agents (전문가들)

하위 메뉴(Dropdown):

Structure Agent (구조 분석)

Toxicology Agent (독성 예측)

IP/Patent Agent (특허 분석)

설정 내용: 각 전문가의 페르소나와 전문 지식 프롬프트.

[Tab 3] 📚 RAG Answer Engine (답변 작성기)

설정 내용: 검색된 골든셋 데이터를 가지고 최종 답변을 작성하는 프롬프트.

"전문 용어를 사용하되, 고등학생도 이해할 수 있게 쉽게 풀어써라."

"답변 끝에는 반드시 참고 문헌(References)을 달아라."

💡 사장님을 위한 한 줄 요약
"Data Operations 메뉴는 '도서관에 책(데이터)을 채우는 곳'이고, AI Tuning 메뉴는 '사서(AI)에게 친절하게 말하는 법을 가르치는 곳'입니다. 그러니 반장(오케스트레이터)과 직원(에이전트) 모두의 교육 매뉴얼이 다 있어야 합니다."
